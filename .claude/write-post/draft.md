# 작성 단계 지침 (write-post 3단계)

## 1. 포스트 초안 작성
`content/posts/<slug>.md` 에 초안을 쓴다. slug는 영문 소문자-하이픈.

파일을 쓰기 전에 `content/posts/<slug>.md`가 이미 존재하는지 확인한다. 이미 존재하면 같은 주제를 다시 다루는 것인지, 슬러그가 겹친 것뿐인지 사용자에게 명확히 알리고, 다른 슬러그를 쓰거나 기존 파일을 덮어쓸지 사용자 확인을 받은 후에만 진행한다. 확인 없이 기존 파일을 덮어쓰지 않는다.

front matter:

    ---
    title: "..."
    date: <현재시각, +09:00>
    description: "..."
    tags: [...]
    draft: true
    source_url: "<입력받은 원문 URL>"
    ---

description 작성 규칙:
- 100자 내외. 이 글을 읽으면 무엇을 알게 되는지 한 문장으로 압축한다.
- "~를 정리합니다", "~에 대해 알아봅니다" 같은 무미건조한 예고형 문장 금지 — 검색 결과·공유 미리보기에 그대로 노출되는 문장이므로, 클릭할 이유가 담겨야 한다.

본문 규칙:
- 비전공자가 이해할 난이도. **일상적인 비유**를 적극 활용(예: 금리를 "이자 온도조절기"에 빗대기).
- 어려운 용어는 짧게 풀어 쓴다.
- 구조: (1) 무슨 일이 있었나 → (2) 왜 중요한가 → (3) 나에게 무슨 의미인가 → (4) 투자 관점에서 보면(§2).
- **두괄식**: 첫 문단(1~2문장)에서 "무슨 일이 있었나"에 곧장 답한다. "~에 대해 알아보겠습니다", "~를 짚어보겠습니다", "~가 화제입니다" 같은 예고형 서론으로 시작하지 않는다 — 검색해서 들어온 독자는 0.5초 안에 "여기 내 답이 있나"를 판단하고, 없으면 바로 이탈한다.
- 서론 직후, 인용블록(`>`)으로 핵심 요약을 2~3문장(무슨 일 + 왜 중요한지)으로 넣는다.
- front matter의 source_url에는 입력으로 받은 원문 기사 URL을 반드시 그대로 기록한다.

## 2. 투자 연관성 (2단계 분석 노트 반영)
포스트 끝에 "투자 관점에서 보면" 단락을 넣는다. analysis.md 단계에서 만든 분석 노트(건드리는 렌즈·선행/동행 구분·확인된 수치·자산군별 함의)를 **쉬운 말로** 풀어 쓴다.

- **세 렌즈로 해석**: 분석 노트의 "건드리는 렌즈"를 짚는다 — 경기 흐름(성장이 살아나나 식나) / 돈의 흐름·유동성(금리·신용 여건) / 기업 체력(실적·밸류에이션).
- **선행 vs 동행 구분**: 분석 노트의 "선행 vs 동행"을 그대로 살려, 지금 당장의 영향인지 앞으로의 경고인지 나눠 설명한다. 타이밍을 뭉뚱그리지 않는다.
- **자산군별 함의**: 분석 노트의 "자산군별 함의"(원가 부담·반사이익 등 구조적 논리)를 이번 뉴스가 어떤 업종·자산군에 유리 또는 불리하게 작용하는지 한두 문장으로 쉽게 풀어 쓴다. 특정 종목이 아니라 업종 단위로 서술한다.
- **사실관계 검증**: 숫자뿐 아니라 모든 사실관계(사건, 원인, 영향 등)에 대해 기사 원문이나 분석 노트의 확인된 자료에만 근거한다. 확인 불가한 사건·정황은 아예 언급하지 않고, 숫자처럼 독자가 구체값을 기대하는 항목만 아래 숫자 레드라인 규칙에 따라 "확인 불가"로 표시한다.
- **숫자 레드라인**: 금리·스프레드·PER 등 구체 수치는 **지어내지 않는다**. 분석 노트에 확인된 값만 쓰고, 없으면 "확인 불가"로 남긴다. "Fed가 다음 달 내린다" 식 **단정 예측 금지** — 필요하면 "시장은 인하 가능성에 무게를 두는 분위기" 정도로만(확인된 범위에서).
- **기준일 병기**: 확인된 수치를 쓸 때는 항상 기준일을 함께 적는다(예: "2026년 7월 기준", "발표 시점 기준"). 기준일이 없는 수치는 분석 노트에도 없다는 뜻이므로 "확인 불가"로 남긴다 — 기준일을 지어내서 채우지 않는다.
- **개인화 조언 금지**: "지금 사라/팔라", 특정 종목 매수·매도 권유는 절대 넣지 않는다. 일반적 맥락 설명에 그치고, 단락 끝에 한 줄 면책을 붙인다: *"이 글은 투자 판단을 돕는 정보일 뿐, 개인 맞춤 투자 조언이 아닙니다."*

## 3. 용어 사전 (필요 시)
본문에 새 핵심 용어가 등장하고 `content/dictionary/_terms.yaml`에 아직 슬러그가 없다면:
1. `content/dictionary/<term-slug>.md` 에 설명 초안을 쓴다(`draft: true`, tags에 "용어사전", description에 그 용어를 한 문장으로 정의).
2. `content/dictionary/_terms.yaml`에 새 블록을 append한다:

   ```yaml
   <term-slug>:
     title: "<용어>"
     aliases: ["<실제 기사에 나올 법한 동의어1>", "<동의어2>"]
   ```

   aliases는 문법적 활용형이 아니라, 실제로 다른 기사에서 이 용어를 다르게 부를 만한 표현 2~3개로 제한한다.

## 4. 위키 링크
1. **용어**: `content/dictionary/_terms.yaml`을 읽어 slug/title/aliases 전체 목록을 얻는다. 본문에서 title 또는 aliases 중 하나와 일치하는 표현이 나오면 상대링크를 건다. 예: `[기준금리](/dictionary/base-rate/)`.
2. **과거 포스트**: `content/posts/`를 스캔해(파일명·front matter title·tags) 이번 뉴스와 주제가 겹치는 과거 글 후보를 찾는다. 관련 있으면 같은 방식으로 링크한다. 예: `[호르무즈·홍해 공급망 이슈](/posts/hormuz-red-sea-oil-supply-shock/)`.
3. `[[...]]` 문법은 쓰지 않는다.
