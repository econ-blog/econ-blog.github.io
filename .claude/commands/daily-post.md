## 0. 모드 계약 (반드시 먼저 확인)
인자를 확인한다.
- **인자 없음 = 무인 모드**: 대화형 도구(AskUserQuestion 등)를 절대 호출하지 않는다. 랭킹 1위 후보를 자동으로 선택한다. 결과는 `auto/post-YYYY-MM-DD` 브랜치에 커밋·푸시하고 PR을 만든다. `main`에 직접 푸시하지 않는다. `git status`로 의도치 않은 파일이 섞이지 않았는지 확인한 뒤, 명시한 경로만 `git add`한다.
- **인자 `manual`= 수동 모드**: 후보 3건을 표로 제시하고 사용자가 고른다. 이후 파이프라인은 동일하되, 마지막에 승인 게이트를 거쳐 `main`에 직접 푸시한다.

이 계약을 이후 모든 단계에 적용한다. 아래 단계 설명은 모드에 따라 갈리는 지점만 표시한다.

## 1. 랭킹
`.claude/daily-post/rank.md`를 Read로 읽고 그 지침에 따라 오늘의 후보를 수집·채점한다.
- 무인 모드: 1위 후보가 8점 미만이면 여기서 종료한다. 브랜치도 PR도 만들지 않는다.
- 수동 모드: 표로 제시된 후보 중 사용자가 1건을 고른다. 3건 전부 8점 미만이면 "오늘 추천 후보 없음"을 보고하고 멈춘다.

## 2. 원문 읽기
선택된 뉴스 1건(제목/원문 URL)에 대해 WebFetch로 원문을 읽어 사실관계(무슨 일이, 왜, 어떤 영향)를 파악한다. RSS에는 본문이 없으므로 반드시 원문을 읽는다.

WebFetch가 실패하거나(차단, 404, 타임아웃, 페이월 등) 원문 내용을 확인할 수 없으면, 제목이나 추측만으로 사실관계를 지어내 초안을 작성하지 않는다.
- 무인 모드: 이 후보를 폐기하고 종료한다(랭킹 2위로 넘어가지 않는다 — 2위는 검증되지 않은 후보라 같은 문제가 반복될 수 있다). 산출물 없음.
- 수동 모드: 사용자에게 원문을 읽을 수 없었다고 알리고 중단한다.

## 3. 연관 기사 수집
WebSearch로 같은 주제를 다룬 다른 한국 경제기사를 3건 안팎 선별한다. 결과는 파일로 저장하지 않고 5단계(draft)로 그대로 이어서 넘긴다 — 5단계가 포스트 front matter의 related_articles 필드에 기록하고, `layouts/partials/related_articles.html`이 그것을 읽어 글 하단에 표시한다.

선별 규칙:
- 입력으로 받은 원문 URL(source_url) 자체는 제외한다.
- 원문 기사와 같은 날 발행된 기사는 제외한다.
- 그보다 앞선 기사, 특히 이번 사건의 배경·맥락을 다룬 예전 기사를 우선한다.
- URL 기준으로 중복 제거. 같은 통신사 기사가 여러 매체에 실린 경우 제목 기준으로도 제거. 같은 날짜에 같은 내용을 다룬 기사가 둘이면 하나만 남긴다.
- 오래된 순으로 정렬한다.
- 각 항목은 title(기사 제목), url, source(언론사 표시명)로 정리한다.
- 검색 결과에 없는 URL을 지어내지 않는다. 실제로 열리는지, 기록한 제목과 같은 기사인지 확인한 것만 넘긴다.
- 3건 안팎을 목표로 하되 꽉 채우려 하지 않는다. 2건뿐이면 2건만 넘긴다.

**WebSearch가 실패하거나 결과가 0건이면** related_articles 없이 계속 진행한다 — 억지로 URL을 지어내지 않는다. 0건은 오류가 아니라 정상 동작이다.

## 4. 분석 단계
`.claude/daily-post/analysis.md`를 Read로 읽고, 그 지침에 따라 이번 뉴스의 투자 관점 분석 노트를 만든다. 분석 노트는 파일로 저장하지 않고 5단계로 그대로 이어서 쓴다.

## 5. 작성 단계
`.claude/daily-post/draft.md`를 Read로 읽고, 그 지침에 따라 포스트 초안(및 필요 시 용어사전 초안)을 파일로 쓴다. 4단계의 분석 노트를 "투자 관점에서 보면" 단락에 반영한다. 슬러그 충돌 처리는 draft.md §1의 모드 분기(무인: 순번 접미사 / 수동: 사용자 확인)를 그대로 따른다 — 지금이 그 모드를 draft.md에 전달하는 지점이다.

## 6. 게시
초안은 **파일로만** 저장한다.

- **무인 모드**: 초안이 저장되면 그것으로 이 실행의 산출물이다. `draft: true`를 그대로 유지한 채, 대상 파일만 명시적으로 나열해 `git add`한다(글롭 금지). `git status`로 다른 변경이 섞이지 않았는지 확인한다. `auto/post-YYYY-MM-DD` 브랜치를 만들어 커밋(`git commit -m "post: <제목> (draft)"`)하고 그 브랜치로 푸시한 뒤, `gh pr create`로 PR을 만든다. `main`에는 어떤 경우에도 직접 푸시하지 않는다.
- **수동 모드**: 사용자에게 (a) 생성한 파일 경로, (b) 본문 요약을 보여주고 "draft:false로 바꿔 main에 푸시할까요?"라는 구체적 질문으로 승인을 요청한다. 이 질문에 대한 명확한 긍정("네", "승인", "푸시해줘" 등)만 승인으로 인정한다. "좋아요", "괜찮네요" 같은 모호한 반응은 승인으로 치지 않고 다시 명확히 묻는다.
  사용자가 명확히 승인하면 그때만: 대상 파일들의 `draft: true` → `draft: false`로 변경 → `git status`로 확인 → 명시한 경로만 `git add` → `git commit -m "post: <제목>"` → `git push origin main`. push는 저장소 전용 배포키로 나간다(ssh config에 등록됨). 승인 전에는 어떤 git 쓰기도 하지 않는다.
